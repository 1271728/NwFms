# 中期答辩材料：系统功能实现与技术细节说明（可直接用于讲解）

> 项目：高校科研经费报销管理系统（FMS）  
> 建议使用方式：答辩时按“总体架构 → 模块功能 → 关键流程 → 现存问题与计划”顺序讲。

---

## 1. 总体技术路线（你可以先讲这 1 分钟）

### 1.1 架构选型
- **前端**：Vue 3 + TypeScript + Vite + Element Plus，负责角色化页面、表单交互、审批操作与统计展示。
- **后端**：Spring Boot + MyBatis，按 Controller / Service / Mapper 分层，负责鉴权、业务规则、流程状态流转与数据持久化。
- **认证鉴权**：Sa-Token，登录后签发 token，以 `userId` 作为登录态标识。
- **数据库**：MySQL，围绕“项目-预算-报销-审批-支付-归档”建模，保证业务闭环。

### 1.2 业务主线
系统核心链路是：
**立项（项目）→ 编制/调整预算 → 发起报销 → 多级审批（组长/二级单位/财务）→ 支付 → 归档 → 看板/报表统计**。

## 2. 每个功能是怎么做的（可直接按模块答辩）

## 2.1 登录认证与权限控制
### 功能目标
- 用户登录、退出、查询个人信息、修改密码。
- 不同角色看到不同菜单、进入不同首页、执行不同审批动作。

### 技术实现
1. **登录**：
   - 后端校验账号状态与密码（兼容占位哈希、MD5、调试明文）。
   - 登录成功后通过 Sa-Token 建立会话并返回 token 与用户角色/权限。
2. **权限来源**：
   - 后端通过 RBAC 查询用户角色、权限码；
   - 前端根据角色动态生成菜单，并在路由守卫中阻止越权访问。
3. **密码修改**：
   - 前端在个人信息抽屉中提交旧密码+新密码；
   - 后端校验旧密码后写入新哈希。

### 你可以强调的点
- 权限控制不是只做前端隐藏，后端 Service 里也做了角色与状态双重校验（防越权）。

---

## 2.2 用户与组织管理（管理员能力）
### 功能目标
- 维护系统用户、角色分配、账号状态、单位信息。

### 技术实现
- 管理端页面负责用户分页、启停用、重置密码、角色绑定。
- 后端提供用户管理相关接口，服务层按管理员角色执行严格校验。
- 组织树用于约束“二级单位审核权限”和业务数据可见范围。

### 你可以强调的点
- 组织结构与 RBAC 结合，形成“角色 + 单位范围”的授权模型，而不是纯角色模型。

---

## 2.3 项目管理（立项流程）
### 功能目标
- 项目创建/编辑/删除、项目成员维护、立项审批流转、状态追踪。

### 技术实现
1. **数据模型**：`project`、`project_member`、`project_audit_log`。
2. **状态机**：草稿 → 待单位审批 → 待财务审批 → 通过/驳回（并支持撤回、结题、停用等状态）。
3. **规则控制**：
   - 仅项目负责人或管理员可编辑草稿/驳回单；
   - 仅对应单位管理员可做单位审批；
   - 仅财务/管理员可做财务审批。
4. **成员管理**：
   - 在允许状态下可增删成员，保证后续报销申请人与项目关系一致。

### 你可以强调的点
- 项目模块是后续预算与报销的基础主数据，审批通过后才进入经费执行阶段。

---

## 2.4 预算总览与预算调整
### 功能目标
- 展示预算科目树、项目预算执行情况；支持预算调整申请并走审批。

### 技术实现
1. **预算总览**：
   - 提供预算科目树接口与项目预算列表接口，支撑多项目对比。
2. **预算调整单**：
   - 调整单支持创建、修改、提交、撤回、删除；
   - 审批流为“二级单位审核 → 财务复核”；
   - 流程通过待办任务表驱动，审批结果同步写日志与消息。

### 你可以强调的点
- 预算调整不是直接改金额，而是走审批流程，确保预算变更可审计、可追溯。

---

## 2.5 报销管理（核心业务）
### 功能目标
- 报销单创建、明细录入、附件上传下载、提交审批、撤回、查看审批轨迹。

### 技术实现
1. **报销单建模**：
   - 主表 + 明细 + 附件 + 审批日志，支持完整单据生命周期。
2. **提交校验**：
   - 校验项目状态、成员权限、预算科目合法性与预算余额；
   - 提交时冻结预算，避免并发超支。
3. **流程审批**：
   - 按节点推进：组长审批（PI）→ 单位审批（UNIT_ADMIN）→ 财务审批（FINANCE）。
4. **资金一致性**：
   - 审批通过后消耗冻结预算；
   - 驳回则释放冻结预算，确保预算账一致。
5. **附件能力**：
   - 支持多部分上传与文件下载，关联到具体报销单。

### 你可以强调的点
- 通过“冻结-核销-释放”机制解决报销并发下的预算一致性问题，这是系统关键技术点。

---

## 2.6 审批中心（统一待办/已办）
### 功能目标
- 审批人统一在一个入口处理报销与预算调整的待办，查看已办与流程日志。

### 技术实现
- 通过工作流任务表统一承载不同业务类型（`REIMB`、`BUDGET_ADJUST`）。
- 审批中心按角色自动过滤待办：
  - 二级单位管理员优先看到单位节点任务；
  - 财务看到财务节点任务；
  - 管理员可全局查看。
- 执行同意/驳回后，路由到对应业务服务，保证“流程动作”和“业务状态变更”一致。

### 你可以强调的点
- 审批中心做的是“统一流程入口”，具体业务规则仍在各自模块服务层中执行，保证职责清晰。

---

## 2.7 消息中心
### 功能目标
- 对提交、通过、驳回等关键节点进行消息提醒，支持未读统计与批量已读。

### 技术实现
- 在流程关键动作中写入消息表（如提交待审、驳回通知、通过通知）。
- 前端消息中心支持分页查询、单条已读、批量已读、未读计数。

### 你可以强调的点
- 消息中心是流程协同层，不直接改变业务状态，但显著提升节点响应效率。

---

## 2.8 支付与归档
### 功能目标
- 对审批完成的报销单进行支付登记，并沉淀归档记录。

### 技术实现
- 支付模块支持创建/更新支付信息、查看支付详情。
- 归档模块支持创建归档记录与查询，配合报销闭环管理。

### 你可以强调的点
- 支付与归档让系统从“审批系统”升级为“全流程财务业务系统”。

---

## 2.9 首页看板与报表统计
### 功能目标
- 展示预算执行、报销统计、流程处理情况，辅助管理决策。

### 技术实现
- 看板接口输出核心统计指标；
- 报表模块提供预算执行和报销统计查询能力；
- 前端按角色展示不同主页视图（管理员、项目负责人、单位审核、财务中心）。

### 你可以强调的点
- 报表层复用了业务主数据与流程数据，避免“统计口径与业务口径不一致”。

## 3. 前后端如何协同支撑功能

### 3.1 前端层
- 路由守卫：未登录拦截、已登录拉取 `me`、按角色限制路由。
- 动态菜单：依据用户角色动态生成模块入口。
- 页面模块：项目、预算、报销、审批、消息、用户管理等视图与 API 解耦。

### 3.2 后端层
- 统一返回结构与异常处理，保证接口稳定性。
- Service 层集中处理业务规则与权限判断；Mapper 层处理 SQL。
- 工作流与消息作为横切能力接入项目、预算调整、报销模块。

### 3.3 数据层
- 通过状态字段 + 审批日志 + 任务表，形成可追踪、可审计、可恢复的业务轨迹。
